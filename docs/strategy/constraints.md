# Constraints

## 目的

このドキュメントは、StreamerHub で新機能や施策を考える際の制約と、意図的にやらないことを明文化するものです。

## 絶対に守るべき制約

### 1. コア価値を薄めない

StreamerHub の中核は、配信を追いやすくすることです。

以下は避けます。

- コア体験と無関係な機能の肥大化
- 初見で価値が見えにくくなる情報の過積載
- 便利さより複雑さが勝つ UI

### 2. 導線を重くしない

どれだけ高機能でも、使い始めるまでが長いと価値体験率が落ちます。

避けること:

- クリック数だけ増える改善
- 初回で理解しづらい設定必須化
- 主要導線の深い階層化

### 3. イベント文脈を見失わない

StreamerHub はイベントや複数視点文脈で強みが出やすいため、単独の配信一覧化に寄りすぎないようにします。

### 4. データだけで完結させない

整理・分析は重要ですが、それ自体が目的ではありません。

- ユーザーの視聴行動につながること
- 実用価値があること
を優先します。

## 現実的な運用制約

### 1. リソース制約

- 一人または少人数で運用する前提がある
- 重い手作業が継続前提の仕組みは避ける
- メンテナンスコストの高い機能は後回しにする

### 2. データ取得制約

- 利用可能な API や取得可能データに依存する
- 音声取得が難しい場合は、タイトル・概要欄・コメントなど代替情報を使う
- 外部サービス仕様変更の影響を受ける可能性がある

### 3. UI/UX 制約

- 初見ユーザーにも伝わる必要がある
- コア視聴者向けの高機能を入れても、入口は簡単であるべき

### 4. マーケティング制約

- 大規模予算を前提にしない
- 継続可能な SNS / コンテンツ運用を中心に考える
- イベント依存施策だけでなく、平時導線も必要

---

## やらないこと

### 1. 価値が説明しにくい複雑機能を先に増やす

まずは既存価値を明確に伝えられることを優先します。

### 2. すべてのユーザー層に同時最適化する

優先ペルソナを決めて、順番に強くします。

### 3. 表面的な数字だけを追う

- PV だけ増やす
- インプレッションだけ増やす
- バズだけ狙って体験が伴わない

これは避けます。

### 4. 一時的にしか回らない運用をコア化する

担当者の気合いが必要すぎる運用は、構造的に弱いです。

---

## 提案却下の基準

以下に当てはまる提案は、基本的に見送るか再設計します。

- コア価値に直結しない
- 説明に長文が必要
- 主要導線を重くする
- 継続運用が難しい
- 成果測定ができない
- イベント時以外で使いどころが薄い

---

## 提案時の最低条件

新しい案は、少なくとも次を満たす必要があります。

1. 誰向けかが明確
2. どの価値を強化するかが明確
3. どう測るかがある
4. 継続運用できる
5. 既存導線を悪化させない
